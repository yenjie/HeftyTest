# Detailed Review

Assumption: `/raid5/data/yjlee/hefty` is the review target as a snapshot of the current working directory on April 3, 2026. This directory is not a Git repository, so this review is against the current tree rather than a diff.

## Findings

1. High: the published package metadata does not describe the runtime the CLI actually requires, and the CLI imports those dependencies eagerly.

Evidence:

- `pyproject.toml:11-13` declares only `numpy`.
- `src/hefty_tm/cli.py:9-19` imports `benchmark_task1` at module import time.
- `src/hefty_tm/benchmark_task1.py:9-18` imports `matplotlib` and `scipy`.
- `src/hefty_tm/static_tmatrix.py:7-9` also imports `scipy`.

Why this matters:

- A clean install of the package can fail before argument parsing, even for lightweight commands like `list-papers` or `summarize-data`.
- This is especially misleading because the project exposes a console entry point in `pyproject.toml:15-16`, which implies a usable installable CLI.

Constructive fix:

- Add `scipy` and `matplotlib` to `project.dependencies`.
- Move the `benchmark_task1` imports out of the module top level in `cli.py` and into the task-1 command handlers so the non-benchmark commands do not need the plotting/fitting stack.

2. High: the fetch path trusts archive member names and can write outside the intended extraction directory.

Evidence:

- `src/hefty_tm/fetch.py:47-60` iterates over tar members and writes files directly.
- `src/hefty_tm/fetch.py:54` computes the destination as `output_dir / member.name.lstrip("./")`.
- `src/hefty_tm/fetch.py:60` writes to that path without checking that the resolved target still lives under `output_dir`.

Why this matters:

- `lstrip("./")` only strips leading dots and slashes. It does not reject embedded traversal such as `anc/../../evil.txt`.
- The current source URLs are trusted arXiv endpoints, but this helper is written as a general-purpose fetch/extract utility. As written, a malformed or hostile archive can escape the extraction root.

Constructive fix:

- Resolve each destination path and verify it remains within `output_dir` before writing.
- Consider centralizing this in a small `safe_extract_member()` helper so future fetch logic cannot bypass the check.

3. Medium: the note stack is no longer a reliable description of what the code actually does.

Evidence:

- `task.md:1` says, “Use one benchmark, not three.”
- `src/hefty_tm/cli.py:50-120` exposes five task-1 style commands: `task1-benchmark`, `task1-publication-faithful`, `task1-publication-locked`, `task1-publication-smoothed`, and `task1-tang-exact`.
- `docs/publication_gap.md:3` says the repo now exposes two distinct task-1 workflows on purpose.
- `docs/code_structure.md:35-40` still documents only `task1-benchmark` in the command list.
- `docs/code_structure.md:121` and `docs/code_structure.md:302-305` state that `phi(r,T)` is not used in the numerical fit path and that the benchmark is not a forward-model spectral fit.
- The current implementation does use `phi`, a forward cumulant model, and spectral penalties:
  - `src/hefty_tm/benchmark_task1.py:1321-1329`
  - `src/hefty_tm/benchmark_task1.py:1336-1357`
  - `src/hefty_tm/benchmark_task1.py:1884-1910`
  - `src/hefty_tm/benchmark_task1.py:7215-7248`
- `README.md:3` says the workflow is described in `agent.md`, but `agent.md:1-3` is only a three-line generic goal statement and does not encode the actual benchmark contract.

Why this matters:

- This repository is explicitly note-driven. The code is not the only product; the notes are meant to steer the work.
- Right now, an agent or collaborator following `task.md`, `agent.md`, or `docs/code_structure.md` will form materially different expectations about the benchmark than someone reading the current implementation.
- That kind of drift is especially risky in a scientific workflow because it blurs which benchmark is considered canonical.

Constructive fix:

- Choose one source of truth for the benchmark contract.
- Update `task.md`, `agent.md`, `README.md`, and `docs/code_structure.md` together, or explicitly mark one of them as historical and point to the canonical workflow note.
- Add a short “last synchronized with code on YYYY-MM-DD” header to the main workflow note.

4. Medium: the tests are tied to one machine-specific absolute path, so they are not portable.

Evidence:

- `tests/test_core.py:224` hard-codes `Path("/raid5/data/yjlee/hefty")`.
- `tests/test_core.py:246` does the same.

Why this matters:

- The suite passes only when the checkout sits at this exact path.
- That blocks straightforward CI setup, local clones into a different directory, and packaging checks in isolated build environments.

Constructive fix:

- Derive the project root from `Path(__file__)`.
- Better yet, make the tests load their fixture data relative to the test file or through helper functions that accept a root path.

5. Medium: the generic headerless table loader is not actually generic; it breaks on valid one-column numeric files.

Evidence:

- `src/hefty_tm/datasets.py:58-64` infers the number of columns from the first two rows.
- For a one-column numeric file, `np.loadtxt()` on two rows produces a one-dimensional array of length 2, so `shape[-1]` becomes `2`, not `1`.
- The subsequent reshape then fails. I verified this locally with a simple one-column file; `load_table()` raised `ValueError: cannot reshape array of size 3 into shape (2)`.
- The current test coverage only exercises a three-column headerless case in `tests/test_core.py:63-69`.

Why this matters:

- The README and module naming present this as a generic ancillary-table loader, not a narrowly tailored helper.
- It happens not to fail on the current fetched tables I sampled under `data/external/arxiv`, but it will fail as soon as a one-column numeric table is introduced.

Constructive fix:

- Infer the column count from the parsed array shape more carefully, e.g. treat one-dimensional samples as one column when each line is scalar.
- Add a regression test for a one-column headerless file.

## Open Questions

- `task1-benchmark` and `task1-publication-faithful` appear to be semantic aliases. I did not mark that as a bug because `run_task1_benchmark()` is already the publication-faithful branch in `src/hefty_tm/benchmark_task1.py:7206`, but the naming still invites confusion when read next to `task.md`.
- The repo currently carries multiple benchmark narratives at once: the original single-benchmark note in `task.md`, the historical reduced-benchmark explanation in `docs/code_structure.md`, and the newer branch split in `docs/publication_gap.md`. It would help to know which of those is intended to be normative going forward.

## Residual Risk Summary

- The green test suite is not currently strong evidence that the package is installation-safe, because it does not check metadata completeness or console-entry import behavior.
- The fetch path has no security-oriented coverage.
- The documentation drift is large enough that future work could easily optimize the wrong benchmark branch while still “following the notes.”
